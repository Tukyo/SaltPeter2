# GameObject

Named scene entities composed from modular components. Game objects are the primary unit of scene authorship — each carries an id, a name, and a collection of components that define its data and behaviour.

Assets are saved and loaded through the export and import subdirectories.

## API

### [`GameObject`](GameObject.ts)
A scene entity built from components. Add [`GameObjectManager`](GameObjectManager.ts) to your scene to use [`Instantiate`](GameObject.ts) and [`Destroy`](GameObject.ts).

```ts
const go = await Nitrate.GameObject.Instantiate(guid, { x: 50, y: 100 });
go?.Destroy();
```

| Method | Description |
|--------|-------------|
| [`static async Instantiate(guid: string, pos: Vec2): Promise<GameObject \| null>`](GameObject.ts) | Resolves a prefab by GUID, spawns it at the given position, and registers it with the GameObjectManager. Returns null if the manager is not initialized or the GUID cannot be resolved. |
| [`Destroy(): void`](GameObject.ts) | Removes this GameObject from the scene and unregisters it from the GameObjectManager. |
| [`AddComponent(Component: new () => T): T`](GameObject.ts) | Creates and attaches a new component of the given type. Returns the new instance. |
| [`GetComponent(Component: new () => T): T \| null`](GameObject.ts) | Returns the first attached component of the given type, or null if none exists. |
| [`RemoveComponent(Component: new () => T): void`](GameObject.ts) | Detaches and removes all components of the given type. |

### [`GameObjectBuffers`](GameObjectBuffers.ts)
Persistent GPU buffers for game object simulation.

`stateBuffer` holds live physics state for every active game object and is
read-written by the [`GameObjectPhysicsPass`](GameObjectPhysicsPass.ts) each frame. `cellBuffer` holds all
cell data packed flat across all game objects and is updated only on spawn or cell loss.

Two separate uniform buffers are used for erase and stamp passes so that both
{@link device.queue.writeBuffer} calls can be issued before the encoder is
submitted without either overwriting the other's mode field.

Created and owned by [`GameObjectPass`](GameObjectPass.ts).


---

### [`GameObjectCellSchema`](GameObjectCellSchema.ts)
Declares the per-cell GPU buffer layout for game object cells.

All cells for all active game objects are packed into a single flat buffer.
Each entry holds the cell's object-local position, back-references to its
owning game object slot, and the permanent material identity fields.
Dynamic per-cell state (temperature, health, lifetime) is NOT stored here —
the stamp pass carries those forward by reading the current simulation textures
at the cell's previous world position each step.

Consumed by [`GameObjectBuffers`](GameObjectBuffers.ts) (buffer sizing) and [`ShaderFactory`](../shaders/ShaderFactory.ts)
(WGSL struct generation) so both sides stay in sync from a single source.


---

### [`GameObjectColliderSchema`](GameObjectColliderSchema.ts)
Declares the per-boundary-point GPU buffer layout for game object collision.

All boundary sample points for all active game objects are packed into a single
flat buffer. Each entry holds the point's position in object-local space.
The owning game object is resolved via the boundaryOffset and boundaryCount
fields in [`GameObjectStateSchema`](GameObjectStateSchema.ts).

Boundary points are generated by [`ColliderGenerator`](../component/ColliderGenerator.ts) from whatever
[`Collider`](../component/definitions/collider/Collider.ts) component is attached — PixelBodyCollider, BoxCollider, or
CircleCollider — and are uploaded once on spawn or rebuilt on collider dirty.

Consumed by [`GameObjectBuffers`](GameObjectBuffers.ts) (buffer sizing) and [`ShaderFactory`](../shaders/ShaderFactory.ts)
(WGSL struct generation).


---

### [`GameObjectCollisionPass`](GameObjectCollisionPass.ts)
Detects collisions between game object boundary cells and occupied sim cells.

One GPU thread per game object slot. Each thread walks the game object's
pre-built boundary points (stored in {@link GameObjectBuffers.colliderBuffer}),
samples the identity texture at each world position, and accumulates a
collision normal from any occupied cells or out-of-bounds hits. The summed
normal is used to reflect the game object's velocity away from the surface;
the updated state is written back to {@link GameObjectBuffers.stateBuffer}.

Must run AFTER the physics pass and BEFORE the stamp pass. At that point
the identity texture contains only sim cells — game object cells were erased
at the start of the frame, so no self-collision filtering is needed.

Created and owned by [`GameObjectPass`](GameObjectPass.ts). Do not call directly.

### [`GameObjectManager`](GameObjectManager.ts)
Tracks all active [`GameObject`](GameObject.ts) instances and manages their lifecycle.

Scenes that contain GameObjects require a GameObjectManager.

```ts
new Nitrate.GameObjectManager();
```

### [`GameObjectPass`](GameObjectPass.ts)
Coordinates all game object GPU passes each simulation step.

On first encounter of a new [`GameObject`](GameObject.ts), its physics state and cell data are
uploaded to the persistent [`GameObjectBuffers`](GameObjectBuffers.ts) on the GPU. After that, no per-step CPU
involvement is needed for physics — the [`GameObjectPhysicsPass`](GameObjectPhysicsPass.ts) integrates velocity and
updates positions directly in {@link GameObjectBuffers.stateBuffer}.

Each sim step dispatches four sequential GPU passes inside the provided encoder:
  1. Erase — clears previous cell positions from the identity and ownership textures.
  2. Physics — integrates velocity; saves prevPos, updates posX/posY in the state buffer.
  3. Collision — reflects velocity away from occupied sim cells using boundary points.
  4. Stamp — reads full cell state (identity, physics, state textures) at previous positions
             and writes it to new positions, carrying all per-cell data forward coherently.

Ownership swap and all ping-pong swaps are handled by [`SimulationManager`](../simulation/SimulationManager.ts) after
this pass submits, consistent with every other pass in the pipeline.

Collider rebuilds remain CPU-side for now; they run when {@link PixelBodyCollider.dirty}
is set and will be driven by GPU cell-loss detection in a later pass.

NOTE: {@link Transform.position} is not written back from the GPU after physics runs.
It reflects the spawn position only. A position readback pass will be added alongside
collision detection to keep it in sync for editor and debug use.

Created and owned by [`SimulationManager`](../simulation/SimulationManager.ts). Do not call directly.


---

### [`GameObjectPhysicsPass`](GameObjectPhysicsPass.ts)
GPU compute pass that integrates Rigidbody velocity and updates game object
positions directly in the persistent {@link GameObjectBuffers.stateBuffer} each simulation step.

Runs between the erase and stamp passes:
  erase reads old positions → physics updates positions → stamp reads new positions.
No CPU involvement in the hot path after initial state upload.

Created and owned by [`GameObjectPass`](GameObjectPass.ts). Do not call directly.


---

### [`GameObjectStateSchema`](GameObjectStateSchema.ts)
Declares the per-game-object GPU state buffer layout.

Each slot holds live physics state for one active game object.
Consumed by [`GameObjectBuffers`](GameObjectBuffers.ts) (buffer sizing) and [`ShaderFactory`](../shaders/ShaderFactory.ts)
(WGSL struct generation) so both sides stay in sync from a single source.
